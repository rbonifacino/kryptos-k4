# A Reproducible Running-Key Candidate Solution for Kryptos K4

## Abstract

This repository presents a fully reproducible, index-aligned cryptanalytic reconstruction of a candidate plaintext for the unsolved K4 section of Jim Sanborn’s *Kryptos* sculpture (1990). This is representative work product associated with private submission of Abstract and Derivation files by Ryan Bonifacino to Sanborn on 18 November 2025. The exact same files were shared to various moderated groups on 19 November 2025, followed by open public distribution—providing a first indexed public timestamp—at 5:59 PM ET on 20 November 2025 on LinkedIn and X. These are identical to the files ultimately submitted to SSRN under author Ryan Bonifacino on 20 November 2025.

This repository includes:

- the official **97-character K4 ciphertext**,  
- a proposed **86-character plaintext** (with **7 null ciphertext positions**),  
- a complete **1–97 index scaffold**,  
- a full **running-key table** with Vigenère derivations, and  
- the consolidated key stream used to transform the ciphertext into plaintext using the standard relation  
  **Key = (Cipher − Plain) mod 26** (A=0…Z=25).

This document contains **no narrative or interpretive commentary**—only the mathematical reconstruction required for independent verification.

- Total characters: **97**

- Cipher rows (excluding header): **31 + 31 + 31 = 93**

## Section 4 — Running-Key Derivation Method (Vigenère Relation)

All key values in this reconstruction use the standard Vigenère decryption relation:

**Key = (Cipher − Plain) mod 26**

with alphabet values assigned as:

A = 0
B = 1
C = 2
…
Z = 25

For each ciphertext position that maps to a plaintext letter, the key letter is computed as:

KeyVal = (CtVal − PtVal) mod 26
Key = Alphabet[KeyVal]

Null positions (those ciphertext indices with no plaintext letter) produce **no key output**.

This method applies uniformly across:

- Row 1 (indices 5–33)  
- Row 2 (indices 36–65)  
- Row 3 (indices 67–93)

All derivations for these indices are shown in Section 5.

## 7.2 — Linear Indexing Model (1–97)

The ciphertext is treated as a single linear stream indexed from **1** to **97**:

- Characters **1–4**: Header **OBKR**  
- Characters **5–33**: “Row 1” plaintext region  
- Characters **34–35**: Structural nulls  
- Characters **36–65**: “Row 2” plaintext region  
- Characters **66**: Structural null  
- Characters **67–93**: “Row 3” plaintext region  
- Characters **94–97**: Terminal end-padding (nulls)

This indexing is used throughout Section 5.

---

## 7.3 — Header Alignment (OBKR)

The sculpture presents K4 with the four-letter header:

O B K R

These do **not** form part of any plaintext.  
In this reconstruction:

- They occupy **Indices 1–4**,  
- They anchor the alignment of the running-key derivations,  
- They determine the entry point into Sections 6.1–6.4.

The first decryptable plaintext letter begins at **Index 5**.

## 7.4 — Row-Break Model Used in the Derivation Tables

The tables in Section 5 use the following row-break placement because:

1. It preserves the official sculpture lineation, and  
2. It localizes plaintext to known, distinct regions:

Row 1: Indices 5–33
Row 2: Indices 36–65
Row 3: Indices 67–93

Indices **34**, **35**, and **66** are treated as **structural nulls** (neither plaintext nor key letters map to them).

This model does not imply the encryption used separate rows internally; it is a purely documentary reconstruction to allow the derivations to be unambiguously expressed.

## 7.5 — Null-Handling Conventions

Three types of non-plaintext positions are explicitly recognized:

1. **Header Null Region** (Indices 1–4).  
2. **Structural Nulls** (Indices 34–35, 66).  
3. **Terminal Null Region** (Indices 94–97).

These characters yield:

- **No plaintext**,  
- **No key letters**,  
- **No running-key derivations**.

This convention guarantees that the derivation tables reflect only indices with meaningful plaintext output.
